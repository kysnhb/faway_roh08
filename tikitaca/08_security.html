Chapter 08

보안 — 최대 리스크

한 줄로 요약하면 이렇다: 설계는 견고한데 그 설계를 지탱하는 전송 계층이 통째로 비어 있다.

최우선 조치 대상
평문 통신 + 원격 JS 실행 = 데이터 유출이 아니라 코드 실행 문제다

usesCleartextTraffic="true"가 설정된 앱이 WebView로 원격 CDN에서 실행 가능한 JavaScript를 받아온다. HTTP 다운그레이드가 성립하는 순간 공격자가 주입하는 것은 가짜 데이터가 아니라 게임 클라이언트 그 자체다. 확정

구조 평가 — 좋은 쪽

✅ 서버 권한형(server-authoritative) 설계다.
재화·확률·전투 결과가 전부 서버에서 결정되고 클라이언트는 재생만 한다. 메모리 조작으로 다이아를 늘리거나 전투 결과를 바꾸는 고전적 치트는 통하지 않는다.
근거: 프로토콜 1,363종 / 밸런스 테이블 클라 부재 / CombatControl에 Math.random 0회 / 모듈 70.2%가 UI 확정

클라이언트 방어 — 사실상 0

방어 기제유무근거
JS 난독화없음uglify 압축뿐
앱 코드 난독화없음com.megagiant.* 56클래스 원명 유지 (R8 non-obfuscating)
루팅 탐지없음Magisk/Frida/Xposed/RootBeer/goldfish 문자열 0건
에뮬레이터 탐지없음동일 — 탐지 문자열 0건
SSL 피닝없음CertificatePinner·커스텀 TrustManager 0건
서명 검증없음GET_SIGNATURES 0건
네이티브 보호없음유일한 .so는 AndroidX DataStore 헬퍼

추가: allowBackup="true" 확정. Play Integrity 클래스는 존재하나 com.megagiant 호출부가 없어 firebase-auth·recaptcha의 전이 의존으로 보인다 추정.

취약점 목록

#취약점심각도근거
1usesCleartextTraffic="true" + WebView가 원격 CDN에서 실행코드(JS)를 로드 — HTTP 다운그레이드 시 임의 JavaScript 주입 성립최상확정 (manifest + MainActivity mWebView)
2게임 서버가 평문 ws:// — MITM 도청·패킷 조작·리플레이높음확정 (DEFAULT_LOGIN_IP=123.59.33.197:8192)
3결제 개시가 평문 http:// — getOrderInfo.php에 price·playerId를 쿼리스트링 전송높음확정 (krPhp="http://159.75.155.107/")
4CDN hot-serve 구조 — 게임 로직이 스토어 심사 밖에서 교체됨. 공급망 공격 시 전 유저 즉시 영향높음확정
5코드 전면 노출 — 모듈 1,210 · 요청 648 · 응답 715개 명칭 평문. 프로토콜 역설계 용이중간확정
6allowBackup="true" — onestore_prefs(user_id·약관플래그) 백업 추출 가능중간확정
7네이티브 영수증 검증이 클라 로컬 — handlePurchase가 로컬 consume 후 token을 JS로 전달중간확정. 서버측 재검증 여부는 미확인
8클라 신고형 전투 종료 — CG_COMBAT_CLIENT_FINISH·CLIENT_PASS_TIME중간확정
9가격표 클라 하드코딩(VKRU_PRICE)낮음표시 왜곡에 그침 (추정)
10중국 인프라 잔존 URL(fengzhangame·QQ) + 디버그 어드민 주소낮음미사용 추정이나 공격면

총평

서버 권한 설계 자체는 견고하다 — 재화·확률·전투 결과를 클라에서 위조할 수 없다.
문제는 그 서버를 믿게 만드는 전송 계층이 통째로 비어 있다는 것이다.

특히 ①번(cleartext + 원격 JS 실행)은 단순 데이터 유출이 아니라 코드 실행 문제이므로 최우선 조치 대상이다.
IL2CPP 대비 클라 개변 난이도는 비교가 안 될 만큼 낮지만, 실제 피해 경로는 "메모리 핵"이 아니라 네트워크 중간자다. 판단

권고 — 우선순위 순

① 전 통신 TLS 강제

ws:// → wss://, 결제 http:// → https://, usesCleartextTraffic=false + Network Security Config로 cleartext 차단.
결제 파라미터를 쿼리스트링에서 POST 바디로 옮기는 것도 여기 포함된다.

이것 하나가 취약점 1·2·3을 동시에 닫는다. 판단

② CDN 무결성

CDN이 서빙하는 JS에 SRI(Subresource Integrity) 또는 서명 검증 적용. WebView 측에서 해시를 검증하고 불일치 시 기동 차단.
공급망 공격과 심사 우회 양쪽을 막는다.

구조를 바꾸지 않고 넣을 수 있는 가장 실효적인 방어다. 판단

③ allowBackup=false

android:allowBackup="false"로 onestore_prefs(user_id·약관 플래그) 백업 추출 차단.
한 줄 수정이며 비용이 사실상 0이다.

우선순위는 낮지만 즉시 적용 가능하다. 판단

질문답
유저 재화가 해킹당할 수 있나?아니오. 서버 권한 구조라 클라 조작으로는 불가능하다. 확정
유저 통신이 도청당할 수 있나?예. 게임 서버가 평문 ws://이고 결제 개시가 평문 http://다. 확정
결제 정보가 노출되나?부분적으로. 카드정보는 원스토어 SDK가 처리하나 price·playerId가 평문 URL 쿼리스트링에 실린다. 확정
게임 코드가 심사 밖에서 바뀔 수 있나?예. 이것이 최대 리스크다. WebView + CDN hot-serve 구조. 확정
공격자가 클라이언트를 통째로 바꿔치기할 수 있나?이론상 가능하다. cleartext 허용 + 원격 JS 로드 + 무결성 검증 없음의 조합. 확정
치트 유저를 탐지할 수 있나?클라 측 수단은 없다. 루팅·에뮬·후킹 탐지가 전무하므로 전적으로 서버 이상탐지에 의존한다. 확정
