데이터 계측 요청서

2026.07.16 작성된 계측 설계안입니다. 문서의 축은 「어떤 이벤트를 남겨서, 그걸로 무엇을 어떻게 볼 것인가」입니다. 본 보고서의 여러 가설을 검증하는 수단이기도 합니다.

본 보고서와의 관계

「방법론 및 한계」에서 밝힌 한계의 상당 부분이 계측 하나로 해소됩니다.

현재는 유저가 어디에서 이탈하는지, 무엇을 이용하지 않는지, 왜 결제하지 않는지를 데이터로 확인할 수 없습니다. 특히 「26레벨 정체 구간이 결제 전환을 저해한다」는 가설은 계측 없이는 검증이 불가능합니다.

02-1

📡현재 상태

Firebase Analytics는 도입되어 있으나, 계측 범위가 제한적입니다.

항목내용상태
SDK 도입Firebase Unity SDK 12.10.1 (Analytics + Messaging), Android/iOS 플러그인 포함완료
사용자 식별UUID 기반 user_id 설정 + 닉네임 사용자 속성완료
구현된 이벤트앱 실행 · 로그인 · 튜토리얼 완료 · 멘토 완료 · 결제 · 아이템 구매 — 6종제한적
결제 이벤트전송 코드가 주석 처리되어 있어 실제 결제가 집계되지 않습니다.미작동
결제 이벤트 규격GA4 표준 규격(currency/value/transaction_id)을 따르지 않아, 주석을 해제하여도 금액이 집계되지 않습니다.규격 불일치
퍼널 이벤트화면 진입/이탈, 상점 탭, 던전 입장, 전투 결과, 가챠, 광고 시청 — 계측된 항목 없음부재

초기화 경합초기화 완료 보장 없이 이벤트 전송이 가능한 구조로, 기동 직후 이벤트 유실 가능성이 있습니다.확인 필요
🔥가장 시급한 사항

현재 매출 데이터가 집계되지 않고 있습니다. 결제 이벤트 전송 코드가 주석 처리되어 있으며, 주석을 해제하더라도 규격이 표준과 달라 금액이 잡히지 않습니다. 두 가지를 함께 수정하여야 합니다.

02-2

🗺️분석 영역 11개

각 영역은 「알고 싶은 것 → 남길 이벤트 → 나오는 리포트 → 판단 기준」 순으로 설계되었습니다.

#영역확인하고자 하는 것현재우선
A길잡이 퀘스트 (초기 이탈)신규 유저가 몇 번 퀘스트에서 이탈하는가완료만P0
B일일 미션어떤 미션이 수행되지 않는가, 보상은 수령되는가없음P0
C던전 이용 현황7종 중 이용되지 않는 콘텐츠, 입장 장벽없음P0
D상점 / 결제결제가 어느 단계에서 이탈하는가, 어느 탭이 기여하는가매출 0P0
E재화 흐름재화가 어디에서 유입되어 어디로 소비되는가없음P1
F성장 (강화 · 제작)성장 루프가 어디에서 막히는가없음P1
G가챠이용률과 결과 만족도없음P1
H광고 시청보상형 광고 완료율없음P1
I출석 / 복귀재방문 주기없음P2
J엔드 콘텐츠고레벨 유저의 콘텐츠 도달률없음P2
K세션 / 리텐션D1/D7/D30자동수집P2
A · B · C가 최우선인 이유

세 영역 모두 현재 완전히 확인이 불가능한 영역이며, 본 보고서의 핵심 발견과 직결됩니다.

  • A(길잡이 퀘스트) — 퀘스트 시작·개별 완료 함수가 이미 존재하나 계측이 없습니다. 현재는 전체 완료만 집계되어, 완주하지 못한 유저가 어디에서 멈췄는지 확인할 수 없습니다. → 26레벨 가설 검증의 전제
  • B(일일 미션) — 미션 타입이 11종이며 자정 리셋·새로고침·보상 버튼이 분리되어 있습니다. 달성과 보상 수령을 분리 계측하면, 「달성률은 높으나 수령률이 낮은」 항목을 찾아낼 수 있습니다 — 이는 밸런스가 아니라 화면 문제이므로 즉시 개선이 가능합니다.
  • C(던전) — 보스 콘텐츠 문제와 요일던전 입장 제한을 정량으로 확인할 수 있습니다. 특히 「입장하려 했으나 횟수 부족으로 거절된 유저 수」는 이탈 지표인 동시에 상품 설계의 근거가 됩니다.
02-3

📐설계 원칙

명명 및 제약

· snake_case, 「영역_동작」 형식

· 이벤트명 40자 / 파라미터 값 100자 / 이벤트당 파라미터 25개

· 열거형은 문자열로 전송 — 숫자로 보내면 리포트에서 의미를 읽을 수 없습니다

고빈도 발화 금지

진행도를 매번 전송하는 설계는 배제합니다. 몬스터 1마리마다 전송하면 이벤트가 폭증하여 한도·비용 문제가 발생합니다.

상태 변화 시점만 기록하고, 총량은 요약 이벤트로 처리합니다.

개인정보

닉네임 파라미터는 전 이벤트에서 제거합니다. 사용자 식별은 user_id(UUID)로 통일합니다.

개인정보 전송 정책 리스크 + 측정기준 슬롯 낭비 + 이미 사용자 속성으로 설정되어 중복입니다.

구현 위치

신규 이벤트는 기존 래퍼에 집중시키고 호출부에 SDK를 직접 노출하지 않습니다.

화면 배선에 로깅 함수를 추가하지 않습니다 — 중복 발화·순서 문제가 발생합니다. 기존 로직 내부에서 호출합니다.

콘솔 설정 — 자주 누락되는 두 가지
  1. 커스텀 측정기준 등록. 신규 파라미터를 콘솔에 등록하지 않으면 리포트에 아예 표시되지 않습니다. 가장 흔한 누락 사항입니다.
  2. 데이터 보관 기간 2개월 → 14개월 변경. GA4 기본값이 2개월이므로 그대로 두면 장기 리텐션 분석이 불가능합니다. 지금 변경하여야 데이터가 축적됩니다.

검수 기준

완료 보고 시 다음을 확인합니다.

  • DebugView에서 각 신규 이벤트가 파라미터 값과 함께 기록되는지
  • 테스트 결제 1건이 금액·통화와 함께 집계되는지 (매출 복구 확인), 취소 시 실패 이벤트가 기록되는지
  • 길잡이 퀘스트 진행 시 퀘스트 번호와 함께 순서대로 기록되는지
  • 미션 달성과 보상 수령이 각각 별개로 기록되는지
  • 요일던전 횟수 0 상태에서 입장 시도 시 거절 사유가 기록되는지
  • 중복 발화 없음 · 파라미터 제약 준수 · 고빈도 발화 없음
  • 릴리즈 빌드 기준 확인 (에디터·디버그 빌드만으로 판정하지 않음)
참고 — 튜토리얼 완료율에 관하여

모바일 게임 튜토리얼 완료율의 업계 벤치마크는 존재하지 않습니다. 널리 인용되는 「30% 완료」는 GameAnalytics 블로그의 가상 예시가 벤치마크로 재유통된 것임을 원문에서 확인하였습니다. 튜토리얼 정의가 게임마다 달라 업계 표준이 성립하지 않습니다.
