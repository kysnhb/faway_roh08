포맷 해설 — BGDatabase 바이너리

역분석으로 확인한 구조입니다. 모든 정수는 little-endian int32, 문자열은 길이(int32) + UTF-8입니다.

파일 헤더 — 버전(7) + GUID(16B) + 애드온 메타 (BGAddonCodeGen, BGAddonLocalization, 원본 경로 Assets/01_Script/DataTable/DataTable.cs)

핵심 발견 — 테이블(meta) 레코드와 필드 레코드가 완전히 동일한 레이아웃을 쓰고, int16 타입코드로 구분됩니다:

int32 marker(=3) | int16 typeCode | guid16 | string name
int32 cfgLen + cfg | byte flag | string annotation | byte[PAD]  (meta 8 / field 18)
int32 blobLen + blob

페이로드 — 고정폭 타입은 blobLen == 행수 × 폭의 raw 배열. 가변 타입은 int32 count + count×(행인덱스, 끝오프셋) + packed — 문자열·관계·해시테이블·중첩이 전부 이 하나의 레이아웃을 씁니다.

타입코드 — 9=Enum, 25=Bool, 28=Double, 29=Float, 31=Int, 34=String, 44=Nested, 45/46=Relation, 47=EntityName, 61=Color, 8=Hashtable, 85=LocalizedString

파서 소스: D:\newgaems\_work\bgdb_parser.py — 스캔/휴리스틱 없는 순차 파서입니다. 모든 meta가 바이트 단위로 정확히 다음 meta의 marker에 착지하며(재동기화 0회), 순회가 EOF 4바이트 앞(제로 터미네이터)에서 끝납니다. 이것이 포맷 해석이 옳다는 가장 강한 증거입니다.
