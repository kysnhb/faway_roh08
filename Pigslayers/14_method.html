IL2CPP 덤프 — 걸림돌과 해결

문제 — 표준 도구인 Il2CppDumper가 “메타데이터 버전 39는 지원하지 않음”으로 실패했습니다. 처음에는 난독화를 의심했습니다. 실존하는 메타데이터 버전은 31까지였기 때문입니다.

검증 — 매직넘버(0xFAB11BAF)는 정상이었고, 문자열이 평문으로 읽혔습니다. 암호화가 아니라는 뜻입니다. 그리고 번들 헤더에서 엔진 버전이 6000.3.9f1(Unity 6.3)로 확인됐습니다.

결론 — v39는 난독화가 아니라 Unity 6.3의 신규 정식 버전이고, Il2CppDumper가 아직 대응하지 못한 것이었습니다. v39를 지원하는 Cpp2IL로 교체해 해결했습니다 (로그: Using actual IL2CPP Metadata version 39).

BGDatabase 파서 — 자체 제작

게임 테이블은 BGDatabase(유니티 유료 에셋)의 바이너리 포맷입니다. 공개 파서가 없어 포맷을 직접 역분석했습니다.

검증 기준 — 파싱 결과가 “그럴듯한지”가 아니라 기계적으로 대조 가능한지로 판단했습니다.

  • 파싱한 필드명·타입을 IL2CPP 덤프의 BGDatabase 코드젠 클래스와 대조 → 게임 테이블 66개 전부 일치
  • 블롭 길이 검산 — 1268 = 4(행수) + 400(오프셋쌍) + 864(문자열) 형태로 모든 필드 검증
  • 내용 검증 — TBL_STAGE 1행이 돼지의 성 I, 레벨 구간이 1–10 → 11–20으로 단조 증가
  • 순회 정합성 — 모든 meta가 바이트 단위로 정확히 다음 meta의 marker에 착지(재동기화 0회)하고, 순회가 EOF 4바이트 앞에서 끝남. 포맷 해석이 옳다는 가장 강한 증거
  • 최종 결과 69/69 테이블 · 545/545 필드 · UNPARSED 0건. 미매핑 enum 0건, 미해결 관계 GUID 0건

단, 두 가지는 “디코딩”이 아니라 “유도”입니다 — ① 중첩(Nested) 컬럼 3개는 디스크에 빈 페이로드라 자식 테이블의 역관계로 채웠고(자식 테이블이 정본), ② 타입 이름은 코드젠에서 왔습니다(타입 크기는 바이너리가 독립 증명).

⚠ 확인하지 못한 것

1. 가격. IAP 가격은 Google Play 콘솔(서버)에 있습니다. APK에는 상품 ID만 있습니다. 따라서 ARPU·매출 추정은 이 리포트 범위 밖입니다.

2. 확률. 뽑기 확률이 클라이언트 테이블에 없습니다 (EquipmentRate 등 일부 드랍률만 존재). 서버 결정으로 추정됩니다.

3. 이벤트 일정. 이벤트 시작/종료 시각은 테이블에 없습니다. 서버(notice.php / PlayNANOO) 제어로 추정됩니다.

4. 런타임 실사용 여부. IL2CPP는 메서드 본문이 네이티브 코드입니다. Cpp2IL의 diffable-cs 출력은 타입·필드·enum·시그니처만 제공하고 로직은 없습니다. 따라서 “문자열/상수가 존재한다”는 사실이지만 “실제로 그렇게 동작한다”는 확정할 수 없습니다. (대표 사례: AdMob 테스트 광고 ID 잔존 — 존재는 사실, 사용 여부는 미확정)

5. 오디오. AudioClip 82개가 sharedassets0.resource에 FMOD FSB5로 스트리밍 분리돼 있어 미추출입니다. 이름 목록만 확보했습니다.

6. 유저 데이터. 오픈 전이므로 리텐션·전환율·이탈 구간 실측치가 없습니다. 이 리포트의 “초반 이탈 위험” 류 언급은 전부 구조에서 추론한 가설이며 검증되지 않았습니다.
